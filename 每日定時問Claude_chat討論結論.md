# 每日定時向 Claude 提問 — 討論結論

> **現況（2026-06-22 合併）：本專案有兩個機制並存**
> - **機制 A — GitHub Actions + `claude -p`（本資料夾的程式）**：每天清晨寄永和區天氣信到 <your-email>@gmail.com，當紀錄／備援。已上線。
> - **機制 B — claude.ai 雲端 routine（`/schedule` Remote）**：每天台灣 05:50 跑最小 prompt「用一句話給我今天的鼓勵」，專責**準時啟動 Claude 互動 5 小時用量視窗**，讓 reset 落在工作時段。詳細設定與限制見 `05_每日定時問Claude.md` 的〈🌅 機制 B〉段。
> - **分工**：B 才是真正達成「視窗對齊工作時段」目的的那個；A 留著當天氣信／備援。
> - ✅ **已驗證成功（2026-06-22）**：B 自動跑會啟動 5 小時視窗（usage 頁重置約 10:30），專案目的達成。

## 需求
做一個程式，每天固定時間自動向 Claude 提問並取得回答。
- 必須**準時執行**
- 自己的電腦**不一定會開著**

---

## 關鍵結論

### 1. 不能直接操作消費者版 Claude App
消費者版的 Claude 聊天 App／網頁沒有公開介面讓外部程式自動送訊息。
實務上只有兩條路：
- **走 Anthropic API**（建議）：穩定、要付費、需要 API 金鑰。
- 用瀏覽器自動化操作網頁版：很脆弱，網頁一改版就壞，**不建議**。

→ **決定：走 Anthropic API。**

### 2. 本機排程器無法保證準時
cron、Windows 工作排程器、Mac launchd 等都依賴作業系統運作，
**電腦關機時不會在排定時間執行**（頂多開機後補跑，但時間就不準了）。

→ 因為要求「準時」且「電腦不一定開著」，**必須放到雲端**（一台一直開著、與自己電腦無關的機器）。

---

## 雲端方案比較（由簡單到完整）

| 方案 | 費用 | 準時度 | 難度 | 備註 |
|------|------|--------|------|------|
| **GitHub Actions** | 免費 | 普通（尖峰可能延遲幾分鐘） | 低 | repo 60 天無活動會被停用；金鑰可放 Secrets |
| Cloudflare Workers Cron / AWS Lambda + EventBridge | 免費額度通常夠 | 高 | 中 | 要寫部署設定 |
| 便宜雲端主機跑 cron | 每月幾美元 | 最高 | 中高 | 最可控，但要自己維護機器 |

→ **初步建議：從 GitHub Actions 開始。** 免費、設定最快，準時度對「每天問一個問題」的用途足夠。
除非「準時」必須精確到某一分鐘，才考慮方案 2 或 3。

---

## 名詞說明：repo（repository，儲存庫）
- GitHub 上**存放程式碼的一個專案空間**（一個 repo = 一個專案）。
- GitHub 的免費排程功能（GitHub Actions）附在 repo 上運作，
  所以要用它，必須先有一個 repo，把程式放進去，再設定排程時間。
